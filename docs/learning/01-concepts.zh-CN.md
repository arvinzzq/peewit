## 1. 什么是 Agent？

聊天机器人发一次消息、收一次回复，结束。Agent 会循环：它可以选择采取行动、观察结果、
决定下一步——直到目标达成。

```
聊天机器人：用户消息 → 模型 → 回复（一次性）

Agent：     用户消息
              → 模型："下一步该做什么？"
              → 执行行动
              → 观察结果
              → 模型："完成了吗？下一步是什么？"
              → 重复，直到目标达成或循环终止
```

**循环**是根本性的区别。代码库里的所有东西，都是为了让这个循环：安全（权限系统）、
可见（追踪）、可持久（会话）、可扩展（工具、技能、适配器）。

## 2. Agent Loop 的五个阶段

OpenClaw 把 agent loop 定义为五个顺序阶段：

```
intake → context assembly → model inference → tool execution → streaming / persistence
```

| 阶段 | 发生了什么 |
|---|---|
| **Intake（接收）** | 收到用户消息或定时触发 |
| **Context Assembly（组装 context）** | 将系统提示、对话历史、工具描述、技能索引打包成发给模型的完整 payload |
| **Model Inference（模型推理）** | 调用模型；收到纯文字回复或工具调用列表 |
| **Tool Execution（工具执行）** | 对每个工具调用：评估权限，执行，收集结果 |
| **Streaming / Persistence（输出 / 持久化）** | 将输出流式返回给适配器；把这一轮保存到会话 |

循环然后回到 Context Assembly 进行下一步——这次消息历史里已经追加了工具结果。

## 3. 权限系统

模型可以请求任何已注册的工具。没有门控，一个有问题的 prompt 就能删文件、泄露密钥、
执行危险命令。

权限系统在每次工具调用**执行之前**进行评估，返回四种决策之一：

| 决策 | 含义 | 谁来处理 |
|---|---|---|
| `allow` | 自动执行 | 循环继续，无需暂停 |
| `ask` | 暂停，询问用户 | 适配器向用户提示；运行等待 |
| `deny` | 拒绝 — 终止运行 | 循环立即发射 `run_failed` |
| `block` | 配置层面永久禁止 | 循环立即发射 `run_failed` |

关键边界：权限包只负责**评估**，不负责**交互**。当决策是 `ask` 时，适配器（CLI 或 Web）
处理用户交互。这让同一套权限逻辑在所有界面上都能工作。

## 4. Context Assembly

模型在调用之间没有记忆。它只知道发送给它的 payload 里有什么。

每次循环步骤，context assembly 重新构建完整 payload：
- 系统提示（指令、工具描述、技能索引、权限指南）
- 对话中所有之前的消息
- 本轮之前所有的工具调用及其结果
- 当前用户消息

这就是为什么 context assembly 在每次迭代都要运行，而不是在开始时运行一次。每一步都
增加了新的观察（工具结果），模型必须在决定下一步行动之前看到这些内容。

成本含义：更多循环步骤 = 每次调用更多 token。Context compaction（汇总旧历史）的存在
就是为了在长会话中控制这个成本。

## 5. 循环何时停止

循环有六个终止条件：

| 条件 | 发射的事件 | 模型能看到吗？ |
|---|---|---|
| 模型返回纯文字回复（无工具调用） | `run_completed` | 是 — 这就是最终答案 |
| 达到步数上限（默认 12） | `run_failed` | 否 |
| 模型 API 错误 | `run_failed` | 否 |
| 权限 `deny` 或用户对 `ask` 说不 | `run_failed` | 否 |
| 规划停顿重试次数耗尽 | `run_failed` | 先收到重试指令，再终止 |
| 工具找不到 / 执行异常 | `tool_failed` → 继续 | 是 — 作为工具结果反馈 |

最后一行揭示了一个重要的不对称：工具层面的失败作为观察反馈给模型。模型可以决定
尝试不同的方法。只有循环层面的失败（步数上限、API 错误、明确拒绝）才直接触发
`run_failed`。

## 6. 规划停顿检测

模型有时会用一段计划文字代替行动来响应：

> "我先读取文件，然后分析结构，最后生成报告。"

这就是规划停顿——模型在叙述它将要做什么，而不是真正去做。如果不纠正，循环可能
无限地产出计划。

检测：运行时扫描响应中的承诺语言（"I'll…"、"let me…"）、计划标题（"Plan:"、
"Steps:"）和列表项。超过 700 字符的响应或包含完成语言（"done"、"fixed"、"found"）
的响应永远不会被标记。

纠正：注入重试指令——"不要重述计划。现在行动：采取第一个可行的工具行动。" 如果
停顿持续超过重试上限，运行以 `run_failed` 终止。

## 7. 关键洞察

**循环很简单；约束不简单。** 去掉事件发射、权限检查和 compaction——核心循环只是：
调用模型，执行工具，重复。代码库里的每一行都在为这个循环添加让它安全和有用的约束。

**`deny` 停止运行；工具错误不会。** 一个自然的假设是任何失败都停止循环。实际设计：
工具错误（工具找不到、执行异常）作为观察返回给模型。模型决定下一步。只有明确的
人或策略拒绝以及基础设施错误才终止运行。对于可恢复的失败，模型是决策者。

**Context 的无状态性是设计选择，不是限制。** 每步重新构建 context 意味着每个循环
步骤都有对所有已发生事件的完整、一致的视图，没有隐藏状态。

**权限和交互是分离的。** 权限包评估；适配器交互。这就是为什么同样的 `deny` / `ask` /
`allow` 逻辑在 CLI、Web 和后台模式下完全相同。

## 8. 复习问题

1. 聊天机器人和 Agent 的根本区别是什么？
   > 聊天机器人发一次 API 调用，返回一次回复。Agent 会循环：选择行动、观察结果、决定
   > 下一步——重复直到目标达成或循环终止。循环是决定性的特征。

2. OpenClaw agent loop 的五个阶段按顺序是什么？
   > Intake → Context Assembly → Model Inference → Tool Execution → Streaming / Persistence。
   > Tool Execution 之后，循环回到 Context Assembly 进行下一步，此时消息历史里已追加了
   > 工具结果。

3. 为什么权限系统必须存在？举一个没有它会出什么问题的具体例子。
   > 模型可以请求任何已注册的工具。没有门控，从网页注入的 prompt 可能指示 agent 删除
   > 文件或把数据发送到外部服务器。权限系统在每次工具调用执行前进行评估，可以拦截、
   > 拒绝或要求人工审批。

4. 为什么 context 在每次循环步骤都要重新组装，而不是复用上一步的？
   > 模型是无状态的——它只知道当前 API payload 里有什么。每次循环步骤都增加了新的
   > 观察（工具结果），模型必须在决定下一步之前看到这些内容。复用旧 context 意味着
   > 模型基于不完整的视图进行推理。

5. 工具调用抛出执行异常。运行是否终止？模型看到什么？
   > 运行**不会**终止。异常被捕获，发射 `tool_failed`，错误文本作为工具结果消息返回给
   > 模型。模型可以选择重试、换工具或报告问题。工具层面的失败是观察，不是循环层面的
   > 失败。

6. 工具调用从权限策略得到 `deny`。模型能看到拒绝信息吗？
   > 不能。`deny` 立即触发 `run_failed`——运行在拒绝消息被组装进对话之前就已经终止。
   > 这是刻意设计的：`deny` 意味着人或策略说了"停止"，而不是"换一种方式试试"。

7. 什么是规划停顿，为什么它对 agent 循环是个问题？
   > 规划停顿发生在模型用计划文字（"我先读取文件，然后分析它……"）来响应而不调用任何
   > 工具的时候。没有检测机制，循环可能无限地产出计划但不采取任何行动。运行时注入
   > 纠正指令，如果停顿持续超过重试上限则终止运行。
